# 全局报错修复与健壮性提升计划

## 1. 摘要 (Summary)
用户反馈“所有功能在正常使用时都会报错”，并且在前端弹出错误提示，且主要发生在交互操作（提交数据等）时。经过对代码库的探索与测试，虽然核心 API 和数据库均能正常运行，但系统缺乏全局的错误捕获与降级机制（Error Boundary），且后端的 `operationLogger` 中存在潜在的作用域变量读取问题（如全局中间件无法获取 `req.params.id`），同时前后端错误状态码的统一处理机制还可以进一步完善。
本计划旨在系统性地修复潜在的报错漏洞，并引入 React 错误边界（Error Boundary）以及更详细的日志捕获机制，彻底解决“所有功能均报错”的痛点。

## 2. 当前状态分析 (Current State Analysis)
- **前端错误处理**：目前的 `src/lib/api.ts` 已经封装了 `FetchError` 并通过 `sonner` 弹出错误信息。但若组件渲染时因数据异常抛出同步错误，将导致整个 React 树崩溃（白屏）。
- **后端中间件**：`api/utils/logMiddleware.ts` 作为一个全局中间件被挂载，但在尝试读取 `req.params.id` 时会得到 `undefined`（因为全局中间件无法访问具体路由的 params），这虽然不会直接导致 500 错误，但在严格模式下可能引发异常，或者导致日志记录失败。
- **状态管理丢失**：页面强依赖 `useStore` 中的 `user` 状态，如果在交互过程中发生页面重载或组件异常卸载，会导致用户状态丢失并强制重定向到登录页。

## 3. 具体修改方案 (Proposed Changes)

### 3.1 引入全局与局部 Error Boundary
- **文件**：`src/components/ErrorBoundary.tsx` (新建)
- **修改内容**：
  - 创建一个标准的 React Error Boundary 类组件，捕获子组件树中的未处理异常。
  - 在发生崩溃时提供友好的降级 UI（“页面遇到了一点小问题”），并提供“重试”或“返回首页”的按钮，避免白屏死机。
- **文件**：`src/App.tsx`
  - 将 `<ErrorBoundary>` 包裹在路由或页面组件外层。

### 3.2 修复后端 operationLogger 的作用域缺陷
- **文件**：`api/utils/logMiddleware.ts`
- **修改内容**：
  - 修复正则匹配和 ID 提取逻辑。使用正则捕获组来提取 URL 中的动态参数（例如 `studentId` 或 `itemId`），而不是依赖 `req.params`。
  - 增加 `try-catch` 块包裹日志记录逻辑，确保即使日志解析失败也**绝对不会**中断正常的业务请求和响应。

### 3.3 增强前端 API 拦截器健壮性
- **文件**：`src/lib/api.ts`
- **修改内容**：
  - 在 `catch` 块中加入更详细的错误分类。针对 401/403 错误，自动触发退出登录或重新验证；
  - 避免在短时间内触发海量的 Toast 错误提示（加入防抖或节流处理）。

### 3.4 补充异常日志输出机制
- **文件**：`api/utils/errorHandler.ts`
- **修改内容**：
  - 确保将导致 500 错误的详细堆栈信息（`err.stack`）写入后端的持久化日志文件中，方便后续运维排查“必现报错”的根本原因。

## 4. 假设与决策 (Assumptions & Decisions)
- **假设**：用户提到的“所有操作都报错”很可能是由于前端渲染时遇到了未捕获的 TypeError，或是后端的某个全局拦截器（如 logger 或 auth）在特定环境下抛出了同步异常。
- **决策**：由于后端 API 本身经过测试可以正常返回数据，因此防御性编程是第一要务。我们不改变现有业务逻辑，而是在其外围包裹安全网（Error Boundary + Logger Try/Catch）。

## 5. 验证步骤 (Verification steps)
1. 启动项目（`npm run dev`），故意在某个组件的渲染中抛出错误，验证 ErrorBoundary 是否能拦截并显示降级 UI，而不是白屏。
2. 调用任意一个包含 ID 参数的 POST 请求（例如给学生加分），验证后端的 `operationLogger` 是否能正确提取到学生 ID 并存入日志表，且不抛出错误。
3. 模拟网络断开或接口返回 500 错误，验证前端 `toast.error` 是否能优雅提示，并在控制台留下便于调试的堆栈信息。
